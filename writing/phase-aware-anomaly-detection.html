Phase-Aware Anomaly Detection on Flight Time Series

2026 · ~12 min read

A steep bank angle is unremarkable during a training turn and alarming during a landing flare. A 700 ft/min descent is exactly right on final approach and a red flag at cruise altitude. Flight telemetry is non-stationary: the statistical signature of normal behaviour shifts continuously with the phase of flight. Any anomaly detector that ignores this context does one of two useless things. It either flags every dynamic maneuver as an outlier, or it raises its tolerance so high that real deviations vanish into the noise.

This note describes the anomaly-detection pipeline I built into Skyris, the flight-debriefing engine for the 360 Training Squadron of the Hellenic Air Force Academy. The core idea is deliberately simple: do not fit one model to the whole flight. Classify each moment into a flight phase, and score it only against the distribution that phase actually implies. The whole thing runs inside the render loop at 60 Hz and costs under 0.1 ms per frame.

Why a single global model fails

Take a feature built from the usual suspects: airspeed, vertical speed, roll, pitch, load factor. Pool every sample from an entire flight and the resulting distribution is multi-modal by construction. Cruise contributes a dense, low-variance cluster. Takeoff, turns, and landing contribute sparse, high-variance tails. A global unsupervised model has no way to know that those tails are expected. It learns a single notion of "typical" that fits none of the phases.

The failure mode is monotonous: the model assigns its highest anomaly scores to exactly the segments where the aircraft is doing the most, which is precisely where an instructor is paying the most attention and least needs a false alarm. The signal drowns.

Segmenting the flight into phases

Before any model runs, each sample is assigned a flight phase. Rather than a learned classifier, I use a small deterministic rule set over smoothed state variables, which keeps the segmentation explainable and reproducible across flights. Three quantities are enough: altitude above ground level, vertical speed, and roll magnitude.

The thresholds are hysteretic. A transition requires the triggering condition to hold for a minimum dwell time rather than a single sample, so the label does not oscillate when a variable hovers near a boundary. The output is a per-sample phase label that partitions the flight into contiguous segments. On synthetic flights where the true phase is known, this cascade reaches 97.3% classification accuracy, with the few errors concentrated at transition points where, for a second or two, altitude has just crossed 5 m or vertical speed is still ramping up.

The feature vector

Each frame is encoded as a compact 7-dimensional vector. The construction is phase-agnostic even though the models that consume it are phase-specific:

f = [ AGL,                 # barometric altitude above ground (m)
      GPS_speed,           # ground speed (m/s)
      |roll|,              # bank angle magnitude (deg)
      pitch,               # pitch angle (deg)
      |vspeed| * 196.85,   # vertical speed (ft/min)
      heading_rate,        # rate of course change (deg/s)
      |a_body| / g ]       # G-load
                           #  ->  f in R^7

These seven capture what an instructor actually watches: where the aircraft is in the vertical, how fast, how hard it is maneuvering, and how much load it is pulling. Vertical speed is scaled to ft/min because that is the unit aviation standards for descent rate are written in, which matters later for landing scoring.

One Isolation Forest per phase

For each phase I train a separate Isolation Forest on the samples belonging to that phase, pooled across a reference corpus of nominal flights. Isolation Forest fits this problem for three reasons. It is unsupervised, which matters because labelled anomalies in training-flight data are scarce and expensive to produce. Its scoring is logarithmic per sample, cheap enough for a real-time loop. And it makes no distributional assumption, so it tolerates the very different feature scales across phases.

The intuition is geometric. An Isolation Forest builds random binary trees by repeatedly picking a random feature and a random split value. Anomalies are few and different, so they sit in sparse regions and get isolated after only a handful of splits. Normal points sit in dense clusters and take many splits to isolate. The path length to isolate a point is therefore the signal.

Formally, for a point x the anomaly score is s(x) = 2 ** ( -E[h(x)] / c(n) ), where E[h(x)] is the mean path length across the trees and c(n) normalises for the expected path length in a tree built from n points. A score near 1 means anomalous, near 0.5 means normal. Training is a loop over phases, each model seeing only the dynamics that are normal for its phase:

models = {}
for phase, group in samples.groupby("phase"):
    clf = IsolationForest(n_estimators=100, max_samples=256)
    clf.fit(normalize(group[FEATURES], phase))   # per-phase z-score
    models[phase] = clf

So a 30-degree bank learned as ordinary by the TURN model never has to coexist with the tight tolerances of the CRUISE model. That separation is the whole point.

Scoring, normalization, and the real-time pipeline

Because the seven features live on wildly different scales (altitude in the thousands of metres, G-load near 1.0), each is standardised with a per-phase z-score before it reaches the forest. The mean and standard deviation for every feature, in every phase, are computed offline from a representative corpus and baked into the binary as compile-time constants:

// per-phase normalization statistics, no runtime file I/O
constexpr float CRUISE_MEANS[7] = { 1200, 65, 2.5, 1.0, 120, 1.5, 1.02 };
constexpr float CRUISE_STDS [7] = {  400, 15, 3.0, 2.5,  80, 2.0, 0.08 };
// ...GROUND, CLIMB, TURN, DESCEND likewise

At inference, every frame walks the same short path: classify the phase, z-score the features against that phase's statistics, route them to that phase's forest, read a score, and map it to a colour. Positive scores stay green; borderline turns amber; genuine outliers red. The editor paints this straight onto the replay timeline so an instructor can jump to any flagged moment.

Scoring a landing

Landing is the most scrutinised phase of primary training, and it deserves treatment beyond a generic outlier score. During the descent and ground phases the pipeline computes three physically grounded sub-scores and combines them, each threshold anchored to aviation standards for the Tecnam P2002-JF.

  • Sink rate at touchdown. Full marks below 120 ft/min, linearly decaying to zero at 360 ft/min. This is the dominant quality indicator, so it carries the most weight.
  • Bank angle at touchdown. Penalised beyond 2 degrees, zero by 8 degrees. A wing-low touchdown is dangerous regardless of how gently the aircraft settles.
  • Vertical G-load at touchdown. The hardness of the arrival, penalised beyond 1.3 g and zero by 2.5 g.
LandingScore = 0.50 * LS_sink
             + 0.25 * LS_bank
             + 0.25 * LS_gload

Tested across four scenarios the weighting behaves the way an examiner would. A textbook landing scores 0.97; a firm one 0.78; a genuinely hard arrival 0.52. The instructive case is the crosswind landing: an excellent 95 ft/min sink rate, but a 4.8-degree bank at touchdown pulls the total down to 0.81. That is correct aviation judgement encoded in arithmetic, a low-bank requirement the sink rate cannot buy back.

Results: the score behaves like an instructor would

Run the pipeline over a full flight and the anomaly score tells a coherent story. It sits firmly positive through straight-and-level cruise, and dips negative in exactly three places: the initial taxi, both 180-degree turns, and the start of the descent. None of these is a mistake. All three present combinations of features that differ sharply from steady flight, which is precisely what the phase models were built to notice.

Because the display score is oriented so that green is normal and red is anomalous, an instructor reads the timeline the way they read a heart monitor. And the sensitivity is graded: a normal turn dips briefly and recovers, a turn flown with excessive bank stays in the red zone longer and deeper, and an abrupt roll input produces a sharp spike at the moment of entry.

The design goal was never a black-box number. It was a score an instructor can point at during a debrief and say: here, on final, the descent rate left the envelope that every other landing stayed inside.

Why it runs in real time

The cost budget is unforgiving: at 60 Hz there is roughly 16.7 ms per frame, most of it spent rendering terrain and the aircraft. The ML pipeline gets a sliver, and it fits. Feature extraction is seven arithmetic operations. Phase classification is three comparisons. Scoring 100 trees of depth 8 is on the order of 800 comparisons. That totals about 48,000 operations per second, roughly 0.5% of the frame budget, completing in under 0.1 ms.

The trick that makes this deterministic is refusing to touch the heap. Both the normalization statistics and the trained forests are stored as compile-time arrays of thresholds and feature indices. Scoring is nothing but integer comparisons and floating-point additions: no dynamic allocation, no file I/O, no virtual dispatch. Every frame costs the same, which is exactly what a render loop needs.

Limitations and where it goes next

The rule-based segmentation is transparent but brittle at unusual profiles such as go-arounds or aborted takeoffs, where phase boundaries blur; a learned phase model with soft assignments would degrade more gracefully. The forests also assume the reference corpus is genuinely nominal, so contamination in the training flights raises the tolerance silently. And the models are trained offline and frozen, so they do not yet adapt to an individual cadet or a specific airframe.

Two of these point at the same fix. A benchmark with known, injected anomalies lets you measure detection instead of merely inspecting it, and lets you tune tolerance against ground truth rather than assumption. That is the motivation behind FLARES, a synthetic-to-real benchmark validated against a real Cessna 172 dataset. The other, online per-cadet adaptation, would let the system distinguish "anomalous versus the expected flight" from "anomalous versus this student", which is the more useful signal for tracking a trainee's progress over weeks.
