Architecture of Skyris: a Real-Time Flight Replay Engine

2026 · ~14 min read

Skyris is a 3D engine I wrote from scratch in C++20 and OpenGL to reconstruct a training flight over the real terrain it happened on. Given a CSV of telemetry from a custom flight recorder, it places the aircraft in a geographically accurate world, replays the flight with faithful attitude and trajectory, and overlays a machine-learning grade on each phase. It runs at a steady 60 Hz on an Apple M1 with 8 GB of RAM, terrain and all.

The reasonable question is why build an engine instead of using Unity or Unreal. The answer is integration. A commercial engine would have handed me terrain, physics, and scripting for free, but the flight-replay subsystem is the reason the whole thing exists, and I wanted it to be a first-class citizen wired into every layer, from the entity system up to the ML pipeline, rather than a plugin bolted onto a closed framework. This note walks through the architectural choices that made that possible.

The layered architecture

Skyris is organised as a set of layers with strict, one-directional dependencies. Nothing above reaches around a layer to touch the one below it. The engine is a static library; the editor is a separate application that links against it and knows nothing of the graphics API underneath.

The payoff of this discipline is portability and testability. Adding a Vulkan backend would mean implementing the platform abstraction classes in a new folder and touching nothing else. The engine could equally drive a headless flight-analysis tool with no editor at all. The event system enforces the decoupling: producers emit events, consumers subscribe, and no module holds a direct reference to another.

Entity-Component-System instead of an inheritance tree

The traditional object-oriented engine builds deep inheritance hierarchies: a GameObject extends an Actor extends a SceneNode. That produces tight coupling, painful extension, and poor cache behaviour, because each object's data is scattered across the heap. Skyris uses an EnTT entity-component-system instead. An entity is just an integer id; a component is a plain struct of data with no logic; a system iterates over entities that carry a given set of components.

In practice this looks like a view over the registry. Adding new behaviour never touches existing systems, it just adds a component and a view:

auto view = registry.view<TransformComponent, MeshComponent>();
for (auto e : view) {
    auto& tc = view.get<TransformComponent>(e);
    auto& mc = view.get<MeshComponent>(e);
    mc.Mesh->Draw(GetWorldTransform(e), mc.Material, (int)e);
}

This is why the flight-replay feature dropped in cleanly. The FlightLogComponent, the terrain, the physics body, and the collider all followed the same pattern: a new component and a new view, no surgery on the core. That open-for-extension property is the central architectural guarantee of the design.

From model space to the screen

Every object in the scene lives in four coordinate systems in turn, and the renderer's job is to march each vertex through them. Understanding this chain is what lets you reason about why something appears where it does.

Attitude is stored as a quaternion throughout, never Euler angles, to sidestep gimbal lock. Skyris is right-handed to match OpenGL and GLM, so no sign flips leak into the math. The lighting model is Cook-Torrance physically based rendering with image-based lighting, and the framebuffer resolves 4x MSAA, because a flight path drawn against terrain shows aliasing immediately without it.

Rendering 111 by 56 kilometres of terrain

The terrain is the real geography the flight took place over: elevation from NASA SRTM, infrastructure from OpenStreetMap, covering roughly 111 by 56 km centred on Dekeleia Air Base. Rendered naively at full resolution that is tens of millions of vertices, impossible at interactive frame rates. A quadtree solves it by making geometric detail proportional to visual importance: nearby tiles get many polygons, distant ones get almost none.

Two details make this robust. First, the bounding box of every node carries a real minimum and maximum height, extracted in constant time from a precomputed elevation mipmap, so tall terrain is not clipped away by a flat guess. Second, frustum culling uses the positive-vertex test against six planes extracted directly from the view-projection matrix, which rejects an entire chunk with six dot products. Buildings, roads, and runways come from OpenStreetMap, triangulated with earcut and extruded, and vegetation is drawn with GPU instancing: one draw call per tree variant renders tens of thousands of trees.

Replaying the flight

The replay system is what justifies the whole engine. The problem it solves is a timing mismatch: the recorder samples at 50 to 200 Hz depending on load, while the render loop runs near 60 Hz, so on any given frame the playback clock falls between two recorded samples. Finding the bracketing pair is a binary search over timestamps; smoothing between them needs interpolation.

The interpolated frame does three things on the same thread, in sequence, because the combined cost is under half a millisecond: it drives the aircraft transform, it feeds the anomaly-detection pipeline, and it updates a procedural spring-damper "bob" whose amplitude is driven by the recorded gyro rate, so the aircraft visibly shivers more in turbulence. The aircraft position is imposed rather than simulated: physics only handles collision, while the recorded IMU and GPS data dictate exactly where the aircraft is.

Where the frame budget goes

At 60 Hz there are about 16.7 ms to spend. Measured on the M1 with full terrain, OpenStreetMap geometry, instanced vegetation, and active replay with ML scoring, it breaks down cleanly.

SubsystemTime / frameShare
Rendering (terrain, OSM, PBR)11.8 ms70.8%
Physics update (Jolt)1.9 ms11.4%
ImGui / ImPlot UI1.1 ms6.6%
Flight replay + interpolation0.3 ms1.8%
ML scoring (100 trees)0.09 ms0.5%
Swap + miscellaneous1.5 ms8.9%

Rendering dominates, as expected for this much terrain, and everything that makes Skyris a debriefing tool rather than a viewer, the replay and the ML, together costs under 2.5% of the budget. Physics runs on a fixed 1/60 s timestep with Jolt, because a variable step feeds the semi-implicit Euler integrator inconsistent time deltas and produces unstable results.

The choices I would defend, and the ones I would revisit

The two decisions that paid off most were building a custom engine and choosing ECS over inheritance. The first let flight replay reach into every layer; the second made every later feature an additive change rather than a refactor. Blocking, synchronous events kept state management simple at the cost of a heavy handler being able to stall a frame, a fair trade for this application.

What I would revisit: the quadtree rebuild runs on the main thread and can stall for a few milliseconds on large scenes, which double-buffering on a background thread would eliminate. Terrain chunks are regenerated from scratch on every scene load, so caching them to disk would cut load time from around ten seconds to under one. And the whole render pipeline is single-threaded; a Vulkan backend with multi-threaded command buffer recording is the natural next step, and the platform abstraction layer was designed precisely so that swapping it in would leave the renderer, scene, and editor untouched.
